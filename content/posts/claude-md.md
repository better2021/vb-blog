# AI 工程规范

## 最高原则
- 需求有歧义必须停止提问，禁止假设
- 先输出方案确认，再编码
- 只实现当前需求，禁止扩展

## 执行流程
① 理解需求（有歧义→停止提问）
② 修改任务须先读懂现有调用链和风格
③ 输出方案→等确认→编码
④ 自检：无调试代码 / 无硬编码密钥 / 错误处理完整 / 测试覆盖 / 注释到位
⑤ 列出每处改动对应的需求点

## 编码规范
- 函数单一职责，职责超一个必须拆分
- 命名清晰表达意图，禁止无意义缩写
- 字面量提取为具名常量并注释含义
- API/DB/IO 必须有错误处理
- TS 禁止 any；Python 必须类型注解

## 修改规则
- 只改需求涉及的代码，范围最小化
- 保持原有风格（缩进/命名/import 顺序）
- 范围外问题用 TODO: 标注，不擅自修改

## 注释规范
- 文件顶部：用途和所属模块
- 函数/类：功能、参数、返回值、异常
- 复杂逻辑：说明"为什么"而非"做什么"
- 禁止注释重复代码含义

## 测试规范
- 新功能必须附带测试
- 覆盖：正常路径 / 边界条件 / 异常处理
- 命名格式：test_<场景>_<预期结果>

## 安全规范
- 禁止硬编码密钥，使用环境变量
- 用户输入必须：类型检查 + 长度限制 + 格式验证
- 依赖版本必须锁定

## README.md 必须包含
1. 项目简介（一句话）
2. 技术栈列表
3. 本地启动步骤（逐条命令）
4. 目录结构说明
5. 环境变量说明

## Git 提交
格式：`<feat|fix|docs|refactor|test|chore>(<scope>): <描述>`

## 红线（禁止）
- 破坏性操作未二次确认
- 修改需求范围外的代码
- 提交 console.log / print
- 关键功能无测试提交
- 需求模糊时假设直接实现
