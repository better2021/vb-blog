# Codex 全局工作规范

## 项目构建原则

在开始任何编码任务前，先完整理解需求，制定清晰的实现方案，再动手写代码。
遇到不确定的需求，主动提问而不是自行假设。

---

# Cursor System Prompt（工程规范）

## 1. 基本原则
- 先理解需求，不清楚必须提问，禁止假设  
- 先给实现思路，再开始编码  
- 只做当前需求，禁止扩展  
- 每一处改动必须可追溯到需求  

---

## 2. 编码规范
- 单一职责：函数 >40 行必须拆分  
- 命名清晰可读，避免模糊命名  
- 禁止魔法数字，必须定义常量  
- 外部调用（API/DB/IO）必须有错误处理  
- TS 禁止 any，Python 必须类型注解  

---

## 3. 修改规则
- 仅修改必要代码，禁止顺手优化  
- 保持原有代码风格  
- 不修改无关代码、注释、结构  
- 仅清理本次改动产生的无用代码  

---

## 4. 注释规范
必须添加：
- 文件顶部：用途说明  
- 函数/类：功能、参数、返回值  
- 复杂逻辑：解释“为什么这样做”  
- 常量：说明含义来源  
- TODO / FIXME：说明原因  

要求：
- 注释简洁，避免重复代码  
- 与项目语言一致  

---


---

## 5.项目结构规范

### 初始化新项目时必须创建
```
项目根目录/
├── README.md          # 项目说明、快速启动、架构概览
├── .gitignore         # 忽略规则
├── .env.example       # 环境变量模板（不含真实密钥）
├── src/               # 源代码
├── tests/             # 测试文件，结构镜像 src/
└── docs/              # 补充文档（可选）
```

### README.md 必须包含
1. 项目简介（一句话）
2. 技术栈列表
3. 本地启动步骤（逐条命令）
4. 目录结构说明
5. 环境变量说明

---

## 6.代码质量要求

### 编写代码时
- **单一职责**：每个函数只做一件事，超过 40 行考虑拆分
- **命名自解释**：变量、函数名能直接表达意图，无需额外注释
- **错误处理**：所有外部调用（API、DB、文件 IO）必须有错误处理
- **类型安全**：TypeScript 项目禁止使用 `any`；Python 项目添加类型注解
- **避免魔法数字**：所有字面量提取为具名常量

### 修改已有代码时
- 只改动被要求修改的部分，不顺手"优化"无关代码
- 不给未修改的代码添加注释或类型注解
- 保持与原有代码风格一致

---

## 7.测试规范

- 新增功能必须附带测试用例
- 测试文件命名：`{模块名}.test.{ext}` 或 `test_{模块名}.{ext}`
- 每个测试用例用中文（或英文）清晰描述测试场景
- 覆盖正常流程、边界条件、异常情况三类用例

---

## 8.Git 提交规范

提交信息格式：`<类型>: <简短描述>`

| 类型 | 用途 |
|------|------|
| `feat` | 新功能 |
| `fix` | Bug 修复 |
| `docs` | 文档更新 |
| `refactor` | 重构（不改变行为） |
| `test` | 测试相关 |
| `chore` | 构建/工具配置 |

---


## 9. 安全规范
- 禁止硬编码密钥  
- 使用环境变量  
- 所有用户输入必须校验  
- 锁定依赖版本  

---

---


## 10. 禁止行为
- 未确认执行破坏性操作  
- 修改无关代码  
- 提交调试代码（console.log / print）  
- 无测试提交关键功能  

---

## 11. 执行流程
1. 理解需求（不清楚先提问）  
2. 输出方案（简要说明思路）  
3. 编写代码（遵循规范）  
4. 自检验证（确保可运行）  
5. 说明改动（列出修改点）
